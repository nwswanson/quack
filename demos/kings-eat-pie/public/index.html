The Pie Locking Problem

Websocket handlers can arrive at the same time. If two browsers both say "ready" while the backend is reading and writing the same room object, one update can stomp the other. That is how a perfectly good pie room becomes a haunted spreadsheet.

This demo makes the race visible. Every browser joins the same room, then everyone presses Say go. The backend has to decide when the whole room is ready and produce one bite order. Without coordination, concurrent websocket messages could see stale state and start the pie too early, too late, or twice.

The Starlark handler solves it with a host-owned lease lock around the room mutation: 

lock = ctx.locks().acquire(
    "memory:rooms:room:kings",
    ttl_ms = 1000,
    wait_ms = 50,
)

While that lock is held, only one handler updates the room. It records joins, ready flags, and the final pie order, then releases the lock before the next websocket handler takes its turn. The lock is scoped to this site and protected by a fencing token, so an expired old holder cannot accidentally release a newer holder's lease.

The result: browsers can mash the button like a basement-show drum kit, but the backend still walks around the booth in order. When everyone has said go, the pie is eaten exactly once.
